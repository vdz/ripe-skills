5 · Best-practices alignment

Against Anthropic's skill best practices + internal Ripe coherence rules.

  • SKILL.md ≤ 500 lines — all 5 skills compliant (max 406 for routing).
  • Valid YAML frontmatter — all 5 skills have name: and description: properly formed.
  • Descriptions begin with an action verb — "Initializes" / "Creates and modifies" / "Sets up" / "Audit". Good imperative form.
  • Descriptions include "Use when…" guidance — all 5 have explicit "Use when" phrasing.
  • Progressive disclosure (reference files) — 4/5 split. building-ripe-routing is one big SKILL.md with no sub-files.
  • Workflows + checklists — ripe-init has a file-creation checklist; store has a workflow checklist; components has one; routing has one.
  • Examples / templates — ripe-init ships templates; building skills have worked examples; ripe-audit has report template + sample checks.
  • Cross-skill links — mostly wired (one gap; see §4).
  • Cross-skill description pairing — components→routing pair missing in description; routing→store pair implicit, not explicit.
  • Self-contained — no external auth or runtime deps — all skills work with stock RTK 2.x / React 19 / react-redux 9.
  • Internal consistency: src/routes/ vs src/router/ — ripe-init scaffolds src/routes/; building-ripe-routing says src/router/. Same skill family, two file-path conventions. Pick one.
  • Auditor citations resolve — ripe-audit's checklist rule-source citations all point to live sections (post the _staging cleanup).
  • Reference file size discipline — listeners.md is at 620 lines — over Anthropic's general recommendation. Splittable into listener-patterns.md + listener-anatomy.md.
  • Indentation matches Ripe (tabs) — all .md code blocks use tabs.

7 · Gaps — what the system doesn't cover

GAP-1No testing skill

Severity: M · Suggested name: building-ripe-tests

Reducer tests, listener tests, and the test-harness pattern (makeTestHarness, fireEvent.change for selects in jsdom, integration tests crossing branches) are mentioned across creating-a-branch.md, the ripe-init scaffolds, and the demo-park feedback doc — but there's no canonical skill for "how do I test in Ripe?".

What it would cover: test-harness pattern, mocking dispatch + getState, asserting reducer transitions, asserting listener flows (chain of dispatches), component tests with RTL + harness, the vi.resetModules() + dynamic-import trick for once-only listeners.

If we want it: tracer-bullet skill — start small (one SKILL.md + one patterns.md) and grow as projects accumulate test idioms.
GAP-2No "overview" / "introduction" skill for first-time agents

Severity: L · Suggested name: ripe-overview or ripe-introduction

An agent dropping into a Ripe project for the first time has no entry-point skill that explains the system: three layers (store / listeners / components), the application information cycle, the cardinal rules at a glance. It's all in README.md at the repo root, but the agent doesn't auto-load README.

Possible answer: a small ripe-overview skill whose entire purpose is to load when the agent says "what's Ripe?" or when the user asks an architectural question. Could just re-export the README's key sections.

Lighter alternative: add a "How Ripe Fits Together" section to ripe-init/SKILL.md that gets read at project boot. New agents naturally hit ripe-init first; existing projects don't.
GAP-3No migration skill for non-Ripe → Ripe

Severity: L · Suggested name: ripe-migrate (future, when needed)

Currently Ripe is greenfield-first. If an existing React + Redux project wants to adopt Ripe, there's no skill that walks through the transition. Not pressing — wait until the first concrete migration request lands.

GAP-4No atomic-primitives skill (future component library)

Severity: L · Suggested name: building-ripe-primitives (future)

The atomic-primitive carve-out (no-useState exception) is documented in patterns.md but currently theoretical. When the first Ripe shared-component library lands, that's the moment to add a skill for it. Premature today.

8 · Findings — ranked

H — High priority

H1Inconsistency: ripe-init scaffolds src/routes/, building-ripe-routing mandates src/router/

Affects: ripe-init + building-ripe-routing · The same issue was flagged in the demo-park 2026-05-20 audit (H1)

ripe-init/SKILL.md checklist line 36–38 creates src/routes/types.ts, src/routes/routes.tsx, src/routes/router.ts. But building-ripe-routing/SKILL.md says "All routing config lives in src/router/". A fresh project is born inconsistent.

Fix: change ripe-init to scaffold src/router/ (and update app-templates.md). One canonical name; router reads better in lstv-portal-style monorepo too.
H2building-ripe-components description missing routing pair

Affects: discoverability of building-ripe-routing when starting from a component task

Current description ends: "For full features requiring both state and UI, pair with building-ripe-store." A user asking to build a clickable card that navigates won't see routing surfaced.

Fix: change pair phrasing to "For full features, pair with building-ripe-store for state and building-ripe-routing for navigation."
H3building-ripe-routing has no progressive disclosure

Affects: maintenance; one more addition tips past Anthropic's 500-line guideline

406 lines in a single SKILL.md. Comfortable today, fragile tomorrow. The skill covers four distinct topic clusters: router setup, route definitions, the bridge + store branch, hydration patterns + idempotency, programmatic + in-component navigation.

Fix: split into a slim navigator SKILL.md + setup.md + hydration.md + navigation.md. Or at minimum: extract the "Idempotency in setLocation Listeners" + the route-exit listeners into their own route-listeners.md reference file (the patterns most likely to grow next).

M — Medium priority

M1listeners.md at 620 lines — split candidate

Affects: reference-file size; readability

The file covers: file structure, 8 patterns, error handling, where cross-branch listeners live, cross-branch cleanup on entity delete, reading state, action chains, optimistic updates, common mistakes. That's three discrete reading sessions packed into one file.

Fix: split into listener-patterns.md (the 8 patterns + concurrency notes), listener-anatomy.md (file structure, reading state, error handling, common mistakes), and keep optimistic-updates / cross-branch cleanup as their own short anchors. SKILL.md's reference table re-links accordingly.
M2building-ripe-routing description doesn't mention store pair

Affects: discoverability of building-ripe-store when starting from a routing task

The phrase "router-to-store bridge" is in there, which gives an agent a faint hint, but explicit pairing helps.

Fix: append "For full features, pair with building-ripe-store for state and building-ripe-components for UI."
M3ripe-audit description has narrow trigger surface

Affects: discoverability when user asks generically about code quality

Trigger word "audit" hits cleanly. But "review my Ripe code", "is this following the patterns?", "maintainability check" might route to the generic code-review first.

Fix: expand description to include phrases like "Ripe compliance review", "maintainability check", "check this against The Ripe Method patterns".
M4GAP-1 — no testing skill (re-listed)

Affects: completeness of the system

See §7 GAP-1. M severity because testing guidance is genuinely missing; deferring is OK while patterns settle.
